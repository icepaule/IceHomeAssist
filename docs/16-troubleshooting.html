Troubleshooting

Diese Seite sammelt häufige Probleme und deren Lösungen für die IceHomeAssist-Installation. Die Einträge sind nach Kategorie geordnet und enthalten konkrete Befehle zur Diagnose und Behebung.

1. Docker Container startet nicht oder stürzt ab

Symptome

  • Home Assistant Web-UI ist nicht erreichbar unter Port 8123
  • docker ps zeigt den Container nicht als "Up" an
  • Container startet und stoppt sofort wieder (Restart-Loop)

Diagnose

# Container-Status prüfen
docker ps -a | grep homeassistant

# Container-Logs anzeigen (letzte 100 Zeilen)
docker logs --tail 100 homeassistant

# Container-Logs live verfolgen
docker logs -f homeassistant

# Detaillierte Container-Informationen
docker inspect homeassistant

# Systemressourcen prüfen
df -h                    # Festplattenplatz
free -m                  # Arbeitsspeicher
docker system df         # Docker Speicherverbrauch

Häufige Ursachen und Lösungen

Ursache Lösung
Festplatte voll docker system prune -a zum Aufräumen alter Images; Log-Dateien löschen
RAM erschöpft Unnötige Integrationen deaktivieren; Recorder-Purge-Intervall kürzen
Korrupte Konfiguration Konfiguration vor dem Start prüfen (siehe YAML-Fehler weiter unten)
Port 8123 belegt ss -tlnp | grep 8123 prüfen; konkurrierenden Prozess beenden
Docker Daemon nicht gestartet sudo systemctl start docker

Container neu starten

# Sanfter Neustart
docker restart homeassistant

# Container stoppen, löschen und neu erstellen
docker stop homeassistant
docker rm homeassistant
docker run -d \
    --name homeassistant \
    --privileged \
    --restart=unless-stopped \
    -e TZ=Europe/Berlin \
    -v /var/lib/homeassistant/homeassistant:/config \
    -v /run/dbus:/run/dbus:ro \
    --network=host \
    ghcr.io/home-assistant/qemux86-64-homeassistant:stable

2. Home Assistant Logs prüfen

Log-Dateien Übersicht

Log Zugriff Inhalt
HA Core Log Web-UI: Einstellungen → System → Protokolle Integrations-Fehler, Automations-Log
Docker Log docker logs homeassistant Start-Fehler, System-Meldungen
home-assistant.log /var/lib/homeassistant/homeassistant/home-assistant.log Detailliertes Application-Log

Log-Level erhöhen für Debugging

# configuration.yaml - Debug-Logging für spezifische Komponenten
logger:
  default: warning
  logs:
    # MQTT Debug
    homeassistant.components.mqtt: debug
    # Alexa Media Player Debug
    custom_components.alexa_media: debug
    # Automatisierungen Debug
    homeassistant.components.automation: debug
    # Template Debug
    homeassistant.helpers.template: debug
    # Allgemeiner HA-Core Debug
    homeassistant.core: info

Hinweis: Debug-Logging erzeugt sehr viele Log-Einträge und kann die Performance beeinträchtigen. Nach der Fehlersuche den Log-Level wieder auf warning zurücksetzen.

3. MQTT-Probleme debuggen

Symptome

  • Tasmota-Sensoren zeigen "unavailable"
  • Sensordaten aktualisieren sich nicht
  • MQTT-Schalter reagieren nicht

Diagnose-Schritte

# 1. Mosquitto Addon Status prüfen
# In der HA UI: Einstellungen > Add-ons > Mosquitto broker

# 2. MQTT-Verbindung testen (von der Host-Konsole)
mosquitto_sub -h localhost -u mqtt_user -P "password" -t "#" -v
# Wenn Nachrichten eintreffen: Broker funktioniert

# 3. Spezifisches Gerät testen
mosquitto_sub -h localhost -u mqtt_user -P "password" \
  -t "tele/Strom-PV/#" -v
# Erwartung: Alle 300 Sekunden eine SENSOR-Nachricht

# 4. Gerät manuell anfragen
mosquitto_pub -h localhost -u mqtt_user -P "password" \
  -t "cmnd/Strom-PV/Status" -m "0"
# Erwartung: Antwort auf stat/Strom-PV/STATUS*

# 5. Tasmota-Gerät direkt prüfen
# Im Browser: http://IP_DES_GERAETS/ aufrufen
# Konsole öffnen und MQTT-Status prüfen

Häufige MQTT-Probleme

Problem Ursache Lösung
Alle Sensoren unavailable Mosquitto Addon gestoppt Addon neu starten
Einzelnes Gerät offline WLAN-Verbindung des Geräts verloren Gerät-IP im Router prüfen; Gerät neu starten
Werte veraltet TelePeriod zu hoch (Standard: 300s) cmnd/DEVICE/TelePeriod 60
Authentifizierungsfehler Passwort in Tasmota stimmt nicht mit Mosquitto überein Credentials in beiden Systemen abgleichen

4. Alexa Media Player - HTTP 500 Fehler

Symptome

  • HTTP 500 Internal Server Error beim Einrichten der Alexa Media Player Integration
  • Login-Prozess schlägt fehl nach Eingabe der Amazon-Credentials
  • Fehlermeldung enthält "aamation" oder Captcha-Hinweis

Ursache

Der HTTP 500 Fehler wird nicht von Home Assistant verursacht, sondern von Amazons Bot-Erkennung. Amazon erkennt den automatisierten Login-Versuch und blockiert die Anfrage. Dies passiert besonders häufig bei Konten ohne 2FA.

Lösung

  1. 2FA auf dem Amazon-Konto aktivieren (wichtigster Schritt!)
    amazon.de → Konto → Anmeldung & Sicherheit → Zwei-Schritt-Verifizierung
  2. Einige Minuten warten und erneut versuchen
    Amazons Rate-Limiting löst sich nach einiger Zeit auf
  3. Im Browser auf demselben Gerät einloggen
    Vor dem HA-Setup: amazon.de im Browser aufrufen und einloggen
  4. VPN deaktivieren
    VPN-Verbindungen werden von Amazon häufiger blockiert
  5. Integration-Eintrag löschen und neu anlegen
    Einstellungen → Geräte & Dienste → Alexa Media Player → Löschen

5. YAML Syntax-Fehler

Symptome

  • Home Assistant startet nicht nach Konfigurationsänderung
  • Fehlermeldung: "Invalid config" oder "Error loading configuration.yaml"
  • Bestimmte Integrationen oder Entities fehlen

Konfiguration prüfen

# Konfiguration im Container prüfen
docker exec homeassistant python3 -m homeassistant \
    --script check_config -c /config

# Oder über die Web-UI:
# Entwicklerwerkzeuge > YAML > Konfiguration überprüfen

# YAML-Syntax lokal prüfen (auf dem Host)
python3 -c "import yaml; yaml.safe_load(open('/var/lib/homeassistant/homeassistant/configuration.yaml'))"

Häufige YAML-Fehler

Fehler Beispiel Lösung
Falsche Einrückung Tabs statt Leerzeichen verwendet Immer 2 Leerzeichen pro Ebene verwenden, keine Tabs
Fehlender Doppelpunkt name "PV Leistung" statt name: "PV Leistung" Doppelpunkt nach jedem Schlüssel
Sonderzeichen in Strings name: Küche: Steckdose Werte mit Sonderzeichen in Anführungszeichen setzen
Doppelte Schlüssel Zwei mqtt:-Blöcke in derselben Datei Alle Einträge unter einem Schlüssel zusammenfassen
Fehlende !secret-Referenz password: !secret mqtt_pw (Schlüssel existiert nicht) Schlüssel in secrets.yaml prüfen
Template-Syntax {{ states('sensor.xyz) }} (fehlendes Anführungszeichen) Jinja2-Syntax sorgfältig prüfen

Best Practice: Backup vor Änderungen

# Immer ein Backup erstellen, bevor Konfigurationsdateien geändert werden
sudo cp /var/lib/homeassistant/homeassistant/configuration.yaml \
       /var/lib/homeassistant/homeassistant/configuration.yaml.bak

# Nach der Änderung: Prüfen
docker exec homeassistant python3 -m homeassistant \
    --script check_config -c /config

# Bei Fehler: Backup wiederherstellen
sudo cp /var/lib/homeassistant/homeassistant/configuration.yaml.bak \
       /var/lib/homeassistant/homeassistant/configuration.yaml

6. Container-Neustart: Patches gehen verloren

Problem

Änderungen an Python-Bibliotheken innerhalb des Docker-Containers (z.B. unter /usr/local/lib/python3.13/...) gehen bei einem Container-Neustart oder Update verloren. Dies betrifft Hotfixes für Bibliotheken, die direkt im Container gepatcht wurden.

Warum passiert das?

Docker-Container sind ephemeral (kurzlebig). Nur Dateien in gemounteten Volumes (/config) überleben einen Neustart. Alles andere wird aus dem Image neu erstellt.

Lösung

# Option 1: Patch als Startup-Script
# Erstelle ein Script, das beim Container-Start ausgeführt wird:

# /var/lib/homeassistant/homeassistant/scripts/apply_patches.sh
#!/bin/bash
# Patches, die bei jedem Container-Start angewendet werden müssen
pip install --force-reinstall some-package==1.2.3

# Über shell_command in HA einbinden (nach jedem Neustart manuell ausführen):
shell_command:
  apply_patches: "bash /config/scripts/apply_patches.sh"

# Option 2: Auf offizielles Fix warten
# Prüfe GitHub Issues des betroffenen Pakets
# Community-Fixes sind oft schneller als offizielle Releases

# Option 3: Custom Docker Image erstellen (fortgeschritten)
# Dockerfile:
FROM ghcr.io/home-assistant/qemux86-64-homeassistant:stable
RUN pip install --force-reinstall some-package==1.2.3

Hinweis: Der httpx.AsyncClient() hat einen Standard-Timeout von 5 Sekunden. Das Erstellen eines AsyncClient im async-Kontext kann eine "blocking call"-Warnung auslösen. Dies ist ein bekanntes Problem, kein kritischer Fehler.

7. Datenbank-Probleme

Symptome

  • Home Assistant startet sehr langsam
  • History- und Logbook-Seiten laden nicht
  • Fehlermeldung: "database is locked" oder "database disk image is malformed"
  • Hohe CPU-Last durch SQLite

Diagnose

# Datenbankgröße prüfen
ls -lh /var/lib/homeassistant/homeassistant/home-assistant_v2.db*

# Typische Größen:
# home-assistant_v2.db      500MB - 2GB (abhängig von Recorder-Einstellungen)
# home-assistant_v2.db-shm  einige MB
# home-assistant_v2.db-wal  variabel (Write-Ahead Log)

Lösung: Datenbank bereinigen

# Option 1: Recorder-Purge über HA Service
# Entwicklerwerkzeuge > Dienste:
service: recorder.purge
data:
  keep_days: 7
  repack: true

# Option 2: Datenbank komplett löschen (Verlust aller Historien-Daten!)
docker stop homeassistant
sudo rm /var/lib/homeassistant/homeassistant/home-assistant_v2.db*
docker start homeassistant
# HA erstellt beim Start eine neue, leere Datenbank

# Option 3: Recorder-Konfiguration optimieren
recorder:
  purge_keep_days: 7
  commit_interval: 30
  exclude:
    entity_globs:
      - sensor.*_energy_*      # Hochfrequente Sensoren ausschließen
    event_types:
      - call_service           # Service-Calls nicht loggen

8. Custom Component Updates

Symptome

  • HACS zeigt verfügbare Updates an
  • Nach dem Update funktioniert eine Integration nicht mehr
  • "ModuleNotFoundError" oder "ImportError" nach Update

Sichere Update-Prozedur

# 1. Vor dem Update: Backup erstellen
sudo cp -r /var/lib/homeassistant/homeassistant/custom_components \
           /var/lib/homeassistant/homeassistant/custom_components.bak

# 2. Update über HACS durchführen
# HACS > Updates > [Component] > Update

# 3. Home Assistant neu starten
docker restart homeassistant

# 4. Logs prüfen
docker logs --tail 50 homeassistant | grep -i error

# 5. Bei Problemen: Rollback
docker stop homeassistant
sudo rm -rf /var/lib/homeassistant/homeassistant/custom_components
sudo mv /var/lib/homeassistant/homeassistant/custom_components.bak \
        /var/lib/homeassistant/homeassistant/custom_components
docker start homeassistant

Bekannte Update-Probleme

Component Problem Lösung
Alexa Media Player Re-Authentifizierung nötig nach Update Integration löschen und neu einrichten (mit 2FA)
HACS selbst GitHub API Rate Limit überschritten GitHub Token in HACS hinterlegen
Diverse Breaking Changes in neuer HA Version Release Notes vor HA-Update lesen

9. Netzwerk-Probleme

Symptome

  • Integrationen zeigen "unavailable"
  • Geräte werden nicht gefunden
  • mDNS-Discovery funktioniert nicht
  • SNMP-Sensoren liefern keine Daten

Diagnose

# DNS-Auflösung testen
docker exec homeassistant nslookup amazon.de

# Netzwerkverbindung prüfen
docker exec homeassistant ping -c 3 192.168.1.1

# Ports prüfen
ss -tlnp | grep -E '(8123|1883|8883)'

# Docker Netzwerk-Modus prüfen (sollte "host" sein)
docker inspect homeassistant | grep NetworkMode

# SNMP-Erreichbarkeit testen
snmpwalk -v2c -c public 192.168.1.10 system

# mDNS-Geräte im Netzwerk finden
avahi-browse -art

Häufige Netzwerk-Probleme

Problem Ursache Lösung
mDNS funktioniert nicht Container nicht im Host-Netzwerk --network=host bei docker run verwenden
UniFi nicht erreichbar SSL-Zertifikat abgelehnt In der Integration "SSL-Zertifikat verifizieren" deaktivieren
SNMP Timeout SNMP auf MikroTik nicht aktiviert SNMP auf dem Switch aktivieren (siehe Netzwerk)
Tasmota-Geräte nicht erreichbar WLAN-Kanal oder IP geändert Tasmota Web-UI aufrufen und MQTT-Settings prüfen
Fritz!Box Integration Timeout TR-064 Schnittstelle deaktiviert In der Fritz!Box TR-064 aktivieren (Heimnetz → Netzwerk)

10. Node-RED Probleme

Symptome

  • WiFi-Button-Automatisierungen funktionieren nicht
  • Node-RED Addon startet nicht
  • Flows gehen nach Update verloren

Diagnose und Lösungen

# Node-RED Addon Logs prüfen
# In HA: Einstellungen > Add-ons > Node-RED > Protokoll

# Flows-Datei prüfen
ls -la /var/lib/homeassistant/addon_configs/a0d7b954_nodered/flows.json

# Flows manuell sichern
sudo cp /var/lib/homeassistant/addon_configs/a0d7b954_nodered/flows.json \
        /home/mpauli/flows_backup_$(date +%Y%m%d).json

# Node-RED zurücksetzen (Vorsicht: Alle Flows gehen verloren!)
# 1. Addon stoppen
# 2. flows.json löschen
# 3. Addon starten (erstellt neue leere Flows)
# 4. Backup-Flows importieren
