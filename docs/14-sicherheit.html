Sicherheit

Die Sicherheit der Home Assistant Konfiguration umfasst den Schutz sensibler Daten wie Passwörter, API-Tokens und OAuth-Credentials. Durch eine durchdachte Kombination aus secrets.yaml, .gitignore, Pre-Commit Hooks und einer soliden Backup-Strategie werden versehentliche Offenlegungen verhindert und die Wiederherstellbarkeit gewährleistet.

secrets.yaml - Konzept und Verwendung

Die Datei secrets.yaml ist das zentrale Schlüsselmanagement für Home Assistant. Sie enthält alle sensiblen Werte (Passwörter, Tokens, API-Keys), die in der configuration.yaml und den Package-Dateien über die Referenz !secret eingebunden werden.

Funktionsprinzip

# secrets.yaml (NIEMALS committen!)
# ========================================
# Diese Datei enthält alle sensiblen Daten.
# Sie wird von .gitignore ausgeschlossen.
# ========================================

# Grundlegende Einstellungen
home_latitude: "51.xxxxxx"
home_longitude: "7.xxxxxx"
home_elevation: "85"

# MQTT
mqtt_user: "mqtt_user"
mqtt_password: "sicheres_mqtt_passwort_hier"

# Tibber
tibber_api_token: "xxxxxxxxxxxxxxxxxxxxxxxxxxxxx"

# UniFi
unifi_host: "192.168.1.1"
unifi_username: "admin"
unifi_password: "sicheres_unifi_passwort"

# Fritz!Box
fritzbox_host: "192.168.178.1"
fritzbox_username: "admin"
fritzbox_password: "sicheres_fritzbox_passwort"

# Tractive (Haustier-Tracker)
tractive_email: "email@example.com"
tractive_password: "tractive_passwort"

# PetKit
petkit_email: "email@example.com"
petkit_password: "petkit_passwort"

# SNMP
snmp_community: "public"

# Alexa Media Player
amazon_email: "email@example.com"
amazon_password: "amazon_passwort"

# Zonen (GPS-Koordinaten)
garden_latitude: "51.xxxxxx"
garden_longitude: "7.xxxxxx"
neighbor_left_latitude: "51.xxxxxx"
neighbor_left_longitude: "7.xxxxxx"
neighbor_right_latitude: "51.xxxxxx"
neighbor_right_longitude: "7.xxxxxx"

Verwendung in Konfigurationsdateien

# configuration.yaml - Beispiel für !secret Referenzen
homeassistant:
  latitude: !secret home_latitude
  longitude: !secret home_longitude
  elevation: !secret home_elevation

# packages/network_switches.yaml
sensor:
  - platform: snmp
    host: 192.168.1.10
    community: !secret snmp_community

Wichtige Regeln

  • secrets.yaml darf niemals in ein Git-Repository committet werden
  • Alle Schlüssel müssen nach einer Neuinstallation manuell ausgefüllt werden
  • Eine secrets.yaml.example mit Platzhaltern kann committet werden, um die Struktur zu dokumentieren
  • Verschachtelte !secret-Referenzen in Package-Dateien funktionieren nur, wenn die secrets.yaml im selben oder übergeordneten Verzeichnis liegt

Pre-Commit Hook für Secrets-Scanning

Als zusätzliche Sicherheitsschicht wird ein Git Pre-Commit Hook verwendet, der vor jedem Commit prüft, ob versehentlich Dateien mit sensiblen Inhalten committed werden sollen.

Hook Installation

# Pre-Commit Hook erstellen
# Datei: .git/hooks/pre-commit

#!/bin/bash
# ========================================
# Pre-Commit Hook: Secrets Scanner
# Verhindert das Committen sensibler Daten
# ========================================

echo "Prüfe auf sensible Daten..."

# Liste der verbotenen Dateien
FORBIDDEN_FILES=(
    "secrets.yaml"
    "secrets.yml"
    ".storage/"
    "flows_cred.json"
    "*.pem"
    "*.key"
)

# Liste der verbotenen Muster in Dateien
FORBIDDEN_PATTERNS=(
    "password:"
    "api_key:"
    "api_token:"
    "client_secret:"
    "PRIVATE KEY"
    "BEGIN RSA"
    "BEGIN EC PRIVATE"
)

# Gestagte Dateien prüfen
STAGED_FILES=$(git diff --cached --name-only)
EXIT_CODE=0

# Dateinamen prüfen
for file in $STAGED_FILES; do
    for forbidden in "${FORBIDDEN_FILES[@]}"; do
        if [[ "$file" == *"$forbidden"* ]]; then
            echo "BLOCKIERT: $file enthält sensible Daten!"
            EXIT_CODE=1
        fi
    done
done

# Dateiinhalte prüfen
for file in $STAGED_FILES; do
    if [ -f "$file" ]; then
        for pattern in "${FORBIDDEN_PATTERNS[@]}"; do
            if grep -q "$pattern" "$file" 2>/dev/null; then
                echo "WARNUNG: $file enthält möglicherweise sensible Daten ($pattern)"
                # Ausnahme für Beispieldateien
                if [[ "$file" != *".example"* ]] && [[ "$file" != *"docs/"* ]]; then
                    EXIT_CODE=1
                fi
            fi
        done
    fi
done

if [ $EXIT_CODE -ne 0 ]; then
    echo ""
    echo "Commit abgebrochen! Bitte sensible Daten entfernen."
    echo "Nutze 'git reset HEAD ' um Dateien aus dem Staging zu entfernen."
    exit 1
fi

echo "Secrets-Scan bestanden."
exit 0

Hook aktivieren

# Hook ausführbar machen
chmod +x .git/hooks/pre-commit

# Test durchführen
git add secrets.yaml
git commit -m "test"
# Ergebnis: "BLOCKIERT: secrets.yaml enthält sensible Daten!"

.storage Verzeichnis

Das .storage Verzeichnis innerhalb der Home Assistant Konfiguration enthält intern verwaltete Daten wie OAuth-Tokens, Integrationseinstellungen, Geräteregister und Benutzerkonten. Diese Daten werden automatisch von Home Assistant verwaltet und sollten nicht manuell bearbeitet oder committet werden.

Wichtige Dateien in .storage

Datei Inhalt Sensibel?
auth Benutzerkonten, Tokens, Refresh-Tokens Ja - sehr sensibel
auth_provider.homeassistant Passwort-Hashes der HA-Benutzer Ja
core.config_entries Alle Integrations-Konfigurationen Ja (enthält oft Tokens)
core.device_registry Registrierte Geräte Nein
core.entity_registry Registrierte Entities Nein
core.area_registry Definierte Bereiche/Räume Nein
lovelace Dashboard-Konfigurationen (UI-Modus) Nein
hacs.* HACS Repositorys und Installationen Nein

Wichtig: Bei einer Wiederherstellung aus dem Git-Repository müssen die Daten aus .storage separat gesichert und wiederhergestellt werden, oder die Integrationen müssen neu eingerichtet werden. Siehe auch Wiederherstellung.

Backup-Strategie

Eine mehrstufige Backup-Strategie stellt sicher, dass die Konfiguration im Fehlerfall wiederhergestellt werden kann.

Backup-Ebenen

Ebene Methode Häufigkeit Inhalt
1. Git Repository GitHub (IceHomeAssist) Bei jeder Änderung YAML-Konfiguration, Scripts, Dokumentation
2. HA Snapshots Home Assistant Backup Wöchentlich (automatisch) Komplette Konfiguration inkl. .storage
3. Host-Backup rsync / tar auf externem Speicher Täglich /var/lib/homeassistant/ komplett
4. Offline-Kopie USB-Stick / NAS Monatlich Komplett-Archiv mit secrets.yaml

Automatisches Backup-Script

#!/bin/bash
# backup_ha.sh - Automatisches Home Assistant Backup
# Wird per Cron täglich um 03:00 Uhr ausgeführt

BACKUP_DIR="/mnt/backup/homeassistant"
HA_CONFIG="/var/lib/homeassistant/homeassistant"
DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/ha_backup_${DATE}.tar.gz"

# Verzeichnis erstellen falls nötig
mkdir -p "$BACKUP_DIR"

# Backup erstellen (inkl. secrets.yaml und .storage)
tar -czf "$BACKUP_FILE" \
    -C /var/lib/homeassistant \
    homeassistant/ \
    --exclude='homeassistant/home-assistant_v2.db*' \
    --exclude='homeassistant/deps/' \
    --exclude='homeassistant/__pycache__/' \
    --exclude='homeassistant/tts/'

# Alte Backups entfernen (älter als 30 Tage)
find "$BACKUP_DIR" -name "ha_backup_*.tar.gz" -mtime +30 -delete

echo "Backup erstellt: $BACKUP_FILE"
echo "Größe: $(du -h "$BACKUP_FILE" | cut -f1)"

Cron-Eintrag

# crontab -e
0 3 * * * /home/mpauli/scripts/backup_ha.sh >> /var/log/ha_backup.log 2>&1

Best Practices

Passwort-Sicherheit

  • Verwende für jeden Dienst ein einzigartiges, starkes Passwort
  • Nutze einen Passwort-Manager (z.B. KeePassXC, Bitwarden) zur Verwaltung
  • MQTT-Passwort: Mindestens 16 Zeichen, alphanumerisch
  • Ändere Standard-Passwörter bei allen Geräten (MikroTik, UniFi, etc.)

Netzwerk-Sicherheit

  • Home Assistant nicht direkt ins Internet exponieren - bei Bedarf VPN verwenden
  • MQTT-Broker nur im lokalen Netzwerk erreichbar machen
  • SNMP Community String ändern (nicht "public" verwenden)
  • Separate VLANs für IoT-Geräte und Smart Home einrichten
  • Firewall-Regeln: Nur notwendige Ports öffnen

Git-Sicherheit

  • .gitignore vor dem ersten Commit erstellen und prüfen
  • Pre-Commit Hook aktivieren und testen
  • Bei versehentlichem Commit von Secrets: alle betroffenen Passwörter sofort ändern
  • Git-History rewriten reicht nicht aus - der Secret muss als kompromittiert betrachtet werden
  • Repository auf privat setzen (GitHub)

Container-Sicherheit

  • Docker Container regelmäßig aktualisieren
  • Container-Images nur von vertrauenswürdigen Quellen beziehen
  • Keine unnötigen Ports nach außen freigeben
  • Docker-Volumes mit korrekten Berechtigungen konfigurieren
  • Patches innerhalb des Containers (/usr/local/lib/python3.13/...) gehen bei Container-Updates verloren - dies beachten

Checkliste: Sicherheitsaudit

Prüfpunkt Häufigkeit Beschreibung
Secrets in Git prüfen Bei jedem Commit Pre-Commit Hook soll warnen
Backup-Integrität prüfen Monatlich Backup testweise entpacken und prüfen
Passwörter rotieren Halbjährlich Alle Passwörter in secrets.yaml erneuern
HACS Components prüfen Bei jedem Update Changelogs auf Sicherheitsfixes prüfen
Netzwerk-Scan Vierteljährlich Offene Ports und unbekannte Geräte prüfen
HA-Benutzer prüfen Vierteljährlich Nicht mehr benötigte Accounts deaktivieren
