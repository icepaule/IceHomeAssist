Automatisierungen

Das IceHomeAssist-System verwendet zwei Automatisierungs-Systeme: Die nativen HA-Automatisierungen (4 YAML-Dateien) und Node-RED Flows (für WiFi-Buttons). Diese Seite beschreibt die 4 HA-Automatisierungen im Detail. Für Node-RED siehe die Node-RED-Seite.

Grundkonzepte

Aufbau einer Automatisierung

Jede Home-Assistant-Automatisierung besteht aus drei Hauptkomponenten:

Komponente Beschreibung Beispiel
Trigger Löst die Automatisierung aus Zeitplan, Sensorwert-Änderung, MQTT-Nachricht
Condition Optionale Bedingungen, die erfüllt sein müssen Nur wenn Strompreis unter X Cent, nur tagsüber
Action Aktionen, die ausgeführt werden Gerät einschalten, Benachrichtigung senden, Szene aktivieren 
# Grundstruktur einer Automatisierung
alias: "Name der Automatisierung"
description: "Beschreibung"
trigger:
  - platform: state
    entity_id: sensor.beispiel
condition:
  - condition: numeric_state
    entity_id: sensor.beispiel
    below: 10
action:
  - service: switch.turn_on
    target:
      entity_id: switch.beispiel
mode: single

1. PV Tibber Savings

Diese Automatisierung berechnet, wie viel Geld durch den PV-Eigenverbrauch eingespart wird. Sie vergleicht den aktuellen PV-Ertrag mit dem Tibber-Strompreis und summiert die Einsparungen über den Tag. 

# automations/pv_tibber_savings.yaml

alias: "PV Tibber Einsparungen berechnen"
description: "Berechnet die Einsparungen durch PV-Eigenverbrauch"
trigger:
  - platform: time_pattern
    minutes: "/5"
    # Alle 5 Minuten aktualisieren
condition:
  - condition: numeric_state
    entity_id: sensor.pv_leistung
    above: 0
    # Nur wenn PV-Anlage Strom produziert
action:
  - service: input_number.set_value
    target:
      entity_id: input_number.pv_savings_today
    data:
      value: >
        {{ states('input_number.pv_savings_today')|float +
           (states('sensor.pv_leistung')|float / 1000 * 5/60 *
            states('sensor.tibber_price')|float) }}
mode: single
Logik: PV-Leistung (W) wird in kWh umgerechnet (geteilt durch 1000, multipliziert mit 5/60 für das 5-Minuten-Intervall) und mit dem aktuellen Tibber-Strompreis (EUR/kWh) multipliziert.

2. Strompreis Automationen

Steuert Geräte automatisch basierend auf dem aktuellen Tibber-Strompreis. Bei niedrigen Preisen werden energieintensive Geräte eingeschaltet, bei hohen Preisen werden sie ausgeschaltet. 

# automations/strompreis_automationen.yaml

alias: "Strompreis-basierte Gerätesteuerung"
description: "Schaltet Geräte basierend auf dem aktuellen Tibber-Strompreis"
trigger:
  - platform: state
    entity_id: sensor.tibber_price
    # Reagiert auf jede Strompreisänderung
action:
  - choose:
    # Günstiger Strom (unter 15 Cent/kWh)
    - conditions:
        - condition: numeric_state
          entity_id: sensor.tibber_price
          below: 0.15
      sequence:
        - service: notify.persistent_notification
          data:
            title: "Strompreis niedrig"
            message: "Aktueller Preis: {{ states('sensor.tibber_price') }} EUR/kWh"
    # Teurer Strom (über 30 Cent/kWh)
    - conditions:
        - condition: numeric_state
          entity_id: sensor.tibber_price
          above: 0.30
      sequence:
        - service: notify.persistent_notification
          data:
            title: "Strompreis hoch"
            message: "Aktueller Preis: {{ states('sensor.tibber_price') }} EUR/kWh - Verbrauch reduzieren!"
mode: single

3. Tibber Next 24h

Ruft die Strompreise für die nächsten 24 Stunden von der Tibber-API ab und speichert sie für die Anzeige im Energie-Dashboard und für die Strompreis-Automatisierungen. 

# automations/tibber_next24h.yaml

alias: "Tibber Preise nächste 24h aktualisieren"
description: "Holt die Strompreisvorhersage für morgen (verfügbar ab ca. 13 Uhr)"
trigger:
  - platform: time
    at: "13:15:00"
    # Tibber veröffentlicht die Preise für morgen gegen 13 Uhr
  - platform: time
    at: "00:05:00"
    # Mitternacht: Tageswerte zurücksetzen
action:
  - service: homeassistant.update_entity
    target:
      entity_id: sensor.tibber_prices
  - delay:
      seconds: 5
  - service: input_number.set_value
    target:
      entity_id: input_number.tibber_min_price_today
    data:
      value: "{{ state_attr('sensor.tibber_prices', 'min_price') | float(0) }}"
  - service: input_number.set_value
    target:
      entity_id: input_number.tibber_max_price_today
    data:
      value: "{{ state_attr('sensor.tibber_prices', 'max_price') | float(0) }}"
mode: single

4. Vacuum Schedule

Startet den Dreame Staubsauger nach einem festen Zeitplan. An verschiedenen Wochentagen werden unterschiedliche Räume gereinigt (Rotation). 

# automations/vacuum_schedule.yaml

alias: "Staubsauger Zeitplan"
description: "Automatische Reinigung nach Wochentag-Rotation"
trigger:
  - platform: time
    at: "10:00:00"
    # Starte täglich um 10 Uhr
condition:
  - condition: state
    entity_id: vacuum.dreame_staubsauger
    state: "docked"
    # Nur wenn der Staubsauger in der Ladestation ist
  - condition: time
    weekday:
      - mon
      - wed
      - fri
    # Nur Montag, Mittwoch, Freitag
action:
  - choose:
    # Montag: Wohnzimmer + Küche
    - conditions:
        - condition: time
          weekday:
            - mon
      sequence:
        - service: dreame_vacuum.vacuum_clean_segment
          target:
            entity_id: vacuum.dreame_staubsauger
          data:
            segments:
              - 1  # Wohnzimmer
              - 3  # Küche
    # Mittwoch: Schlafzimmer + Flur
    - conditions:
        - condition: time
          weekday:
            - wed
      sequence:
        - service: dreame_vacuum.vacuum_clean_segment
          target:
            entity_id: vacuum.dreame_staubsauger
          data:
            segments:
              - 2  # Schlafzimmer
              - 4  # Flur
    # Freitag: Komplettreinigung
    - conditions:
        - condition: time
          weekday:
            - fri
      sequence:
        - service: vacuum.start
          target:
            entity_id: vacuum.dreame_staubsauger
mode: single

Scripts

Scripts sind wiederverwendbare Aktionssequenzen, die von Automatisierungen, Dashboards oder manuell aufgerufen werden können. Sie werden in den jeweiligen Packages definiert (z.B. WiFi-Button-Actions in packages/wifi_buttons.yaml). 

# Beispiel: Script in einem Package
script:
  announce_strompreis:
    alias: "Strompreis ansagen"
    description: "Sagt den aktuellen Strompreis über Alexa an"
    sequence:
      - service: notify.alexa_media_marcus_echo_show_8_2nd_gen
        data:
          message: "Der aktuelle Strompreis beträgt {{ states('sensor.tibber_price') }} Euro pro Kilowattstunde."
          data:
            type: "tts"
    mode: single
HA vs. Node-RED: Im IceHomeAssist-System werden die meisten einfachen Automatisierungen (Zeitpläne, Preisabfragen) nativ in HA definiert. Komplexere, event-basierte Logik (WiFi-Buttons mit dynamischer Funktionszuweisung über input_select) wird über Node-RED gesteuert. Mehr dazu auf der Node-RED-Seite.

Automatisierungs-Modi

Modus Beschreibung Verwendung
single Ignoriert neue Trigger, solange die Automatisierung läuft Standard für alle 4 Automatisierungen
restart Startet bei neuem Trigger die laufende Ausführung neu Geeignet für Timeouts/Delays
queued Reiht neue Trigger in eine Warteschlange ein Für sequenzielle Verarbeitung
parallel Führt mehrere Instanzen gleichzeitig aus Für unabhängige Aktionen
